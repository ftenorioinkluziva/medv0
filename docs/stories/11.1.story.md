## Story

**Como** sistema,
**Quero** classificar o tipo de documento após extração e rotear para o fluxo correto,
**Para** que bioimpedância atualize o perfil e exames de sangue gerem análise.

---

## Contexto

Atualmente `persistence.ts` salva `structuredData.documentType` direto do extractor sem routing. Todo documento gera análise. Bioimpedância (composição corporal) contém dados como % gordura, massa muscular, taxa metabólica — que são mais úteis como dados do Medical Profile do que como trigger de análise.

**Pipeline atual:**
1. Upload → extractor (`extractor.ts`) → `SanitizedMedicalDocument`
2. `persistence.ts` → insert `documents` + `snapshots`
3. Trigger análise (via `trigger-living-analysis.ts`)

**Pipeline proposto:**
1. Upload → extractor → `SanitizedMedicalDocument`
2. **Classificação** → `lab_test` ou `body_composition`
3a. `lab_test` → pipeline atual (snapshot + trigger análise)
3b. `body_composition` → atualizar perfil + histórico (Story 11.2)

## Acceptance Criteria

### AC1 — Classificação de documento

**Given** um `SanitizedMedicalDocument` extraído pelo Gemini
**When** `classifyDocument(structuredData)` é chamado
**Then** retorna `'lab_test'` ou `'body_composition'`

**Cenário A — exame de sangue:**
**Given** structuredData com `modules` contendo categorias como "Hematologia", "Bioquímica", "Hormônios", parâmetros como "Hemoglobina", "Glicose", "TSH"
**When** classificado
**Then** retorna `'lab_test'`

**Cenário B — bioimpedância:**
**Given** structuredData com `modules` contendo parâmetros como "Gordura Corporal %", "Massa Muscular", "Taxa Metabólica Basal", "Gordura Visceral", "Água Corporal"
**When** classificado
**Then** retorna `'body_composition'`

**Cenário C — ambíguo:**
**Given** structuredData que não se encaixa claramente em nenhuma categoria
**When** classificado
**Then** retorna `'lab_test'` como default seguro (gera análise)

### AC2 — Heurística de classificação

**Given** a necessidade de classificar documentos
**When** a heurística é implementada
**Then** usa detecção por keywords nos nomes de parâmetros/módulos:
  - `body_composition` triggers: "gordura corporal", "massa muscular", "gordura visceral", "massa óssea", "taxa metabólica basal", "água corporal", "bioimpedância", "body fat", "muscle mass"
  - Se ≥ 3 triggers encontrados → `body_composition`
  - Caso contrário → `lab_test`

### AC3 — Router integrado no pipeline

**Given** o pipeline de upload
**When** um documento é processado
**Then:**
  1. Extração acontece normalmente
  2. `classifyDocument()` determina o tipo
  3. Documento é salvo com `documentType` atualizado pela classificação
  4. Se `lab_test`: trigger análise (fluxo atual)
  5. Se `body_composition`: NÃO trigger análise, retorna indicação de que é body composition

### AC4 — Response diferenciada para o frontend

**Given** o resultado do upload
**When** documento é classificado como `body_composition`
**Then** API retorna `{ type: 'body_composition', message: 'Dados de composição corporal detectados' }`
**And** NÃO retorna `completeAnalysisId` (sem trigger de análise)

**Given** documento classificado como `lab_test`
**When** processado
**Then** comportamento atual: retorna `{ type: 'lab_test', completeAnalysisId }` (ou similar)

### AC5 — Documento salvo com processingStatus adequado

**Given** documento de `body_composition`
**When** salvo no banco
**Then** `documents.processingStatus` = `'completed'`
**And** `documents.documentType` = `'body_composition'`
**And** snapshot é criado normalmente (para histórico)

### AC6 — Testes

**Given** as mudanças
**When** testes são executados
**Then:**
  - Classificação: exame de sangue → `lab_test`, bioimpedância → `body_composition`, ambíguo → `lab_test`
  - Router: `lab_test` dispara análise, `body_composition` não dispara
  - Response: formato diferente para cada tipo

## Dev Notes

### Classifier

```typescript
// src/lib/documents/classifier.ts
const BODY_COMPOSITION_KEYWORDS = [
  'gordura corporal', 'body fat', 'massa muscular', 'muscle mass',
  'gordura visceral', 'visceral fat', 'massa óssea', 'bone mass',
  'taxa metabólica basal', 'basal metabolic', 'água corporal', 'body water',
  'bioimpedância', 'bioimpedance',
]

const THRESHOLD = 3

export type DocumentClassification = 'lab_test' | 'body_composition'

export function classifyDocument(structuredData: SanitizedMedicalDocument): DocumentClassification {
  const searchText = JSON.stringify(structuredData).toLowerCase()
  const matchCount = BODY_COMPOSITION_KEYWORDS.filter(kw => searchText.includes(kw)).length
  return matchCount >= THRESHOLD ? 'body_composition' : 'lab_test'
}
```

### Upload API route

Verificar `src/app/api/documents/upload/route.ts` ou `src/app/app/upload/` para localizar onde o trigger de análise acontece.

### `processingStatusEnum` existente

`processingStatusEnum` em `documents.ts`: `['processing', 'completed', 'failed']`. Para body composition, status é `'completed'` (extração ok, não houve erro). Pode-se adicionar `'profile_update'` ao enum se necessário, mas `'completed'` é suficiente — o tipo do documento (`documentType = 'body_composition'`) já diferencia.

## Scope

**IN:** Classifier function, routing no pipeline, response diferenciada, document type update
**OUT:** Atualização do perfil (Story 11.2), UI de evolução (Story 11.3), classificação por LLM (heurística apenas)

## Complexity

**M** (Medium) — Classifier + routing + response update. ~2.5h estimado.

## Risks

- **LOW:** Heurística por keywords pode classificar incorretamente documentos ambíguos. Mitigação: threshold de 3 keywords + default `lab_test` (seguro).
- **LOW:** Documentos em inglês vs português — keywords cobrem ambos idiomas.
