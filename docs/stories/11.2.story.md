## Story

**Como** paciente,
**Quero** que meus dados de bioimpedância alimentem meu perfil automaticamente,
**Para** que meu Medical Profile reflita composição corporal atualizada sem intervenção manual.

---

## Contexto

Story 11.1 classifica documentos. Quando `body_composition` é detectado, os dados de composição corporal devem atualizar o Medical Profile e ser registrados em histórico para evolução temporal. O `medical_profiles` já tem `bodyFatPercentage` — faltam campos para os demais indicadores de bioimpedância.

**Campos existentes em `medical_profiles`:** `bodyFatPercentage`, `weight`
**Campos faltantes:** `muscleMass`, `visceralFatLevel`, `boneMass`, `basalMetabolicRate`, `bodyWaterPercentage`

---

## Acceptance Criteria

### AC1 — Campos de bioimpedância no Medical Profile

**Given** `medical_profiles` sem campos completos de composição corporal
**When** migration é aplicada
**Then** campos adicionados:
  - `muscleMass` NUMERIC(5,2) nullable
  - `visceralFatLevel` NUMERIC(5,2) nullable
  - `boneMass` NUMERIC(5,2) nullable
  - `basalMetabolicRate` INTEGER nullable
  - `bodyWaterPercentage` NUMERIC(5,2) nullable

**And** perfis existentes mantêm `null` nos novos campos

### AC2 — Tabela `body_composition_history`

**Given** não existe tabela de histórico de composição corporal
**When** migration é aplicada
**Then** existe tabela `body_composition_history` com:
  - `id` UUID PK
  - `userId` UUID FK → `users.id`
  - `documentId` UUID FK → `documents.id` nullable (se veio de upload)
  - `weight` NUMERIC(5,2)
  - `bodyFat` NUMERIC(5,2)
  - `muscleMass` NUMERIC(5,2)
  - `visceralFat` NUMERIC(5,2)
  - `boneMass` NUMERIC(5,2)
  - `bmr` INTEGER
  - `bodyWater` NUMERIC(5,2)
  - `measuredAt` DATE NOT NULL
  - `createdAt` TIMESTAMP DEFAULT NOW

**And** índice em `(userId, measuredAt)` para queries de evolução

### AC3 — Upload de bioimpedância atualiza perfil e histórico

**Given** Story 11.1 classificou documento como `body_composition`
**When** dados são processados
**Then:**
  1. Extrair métricas do `structuredData` (% gordura, massa muscular, etc.)
  2. Atualizar `medical_profiles` do usuário com os valores mais recentes
  3. Inserir registro em `body_composition_history` com todos os valores + `measuredAt` (data do exame)
  4. Atualizar `medical_profiles.weight` se peso estiver presente

### AC4 — Extração de métricas do structuredData

**Given** um `SanitizedMedicalDocument` de bioimpedância
**When** `extractBodyCompositionMetrics(structuredData)` é chamado
**Then** retorna objeto com campos mapeados:

```typescript
interface BodyCompositionMetrics {
  weight?: number
  bodyFat?: number
  muscleMass?: number
  visceralFat?: number
  boneMass?: number
  bmr?: number
  bodyWater?: number
  measuredAt?: string // data do exame
}
```

**And** busca por nome de parâmetro nos `modules[].parameters[]` do structuredData
**And** converte valores string para número (removendo unidades: "25.3%", "54.2 kg")

### AC5 — Feedback visual para o paciente

**Given** upload de bioimpedância processado
**When** o paciente vê o resultado
**Then** mensagem: "Dados de composição corporal atualizados no seu perfil"
**And** link para visualizar perfil atualizado
**And** NÃO mostra "Análise em andamento" (não houve trigger de análise)

### AC6 — Testes

**Given** as mudanças
**When** testes são executados
**Then:**
  - Schema: campos adicionados em `medical_profiles`, tabela `body_composition_history` criada
  - Extração: `extractBodyCompositionMetrics` parseia corretamente valores com unidades
  - Perfil: campos atualizados após upload de bioimpedância
  - Histórico: registro inserido com valores corretos
  - Peso: `medical_profiles.weight` atualizado se presente

### Parsing de valores

Valores no structuredData podem vir como:
- `"25.3%"` → `25.3`
- `"54.2 kg"` → `54.2`
- `"1.430 kcal"` → `1430`
- `"12"` → `12`

Regex: `/[\d.,]+/` → parseFloat após remover separador de milhar.

### Schema `body_composition_history`

```typescript
// src/lib/db/schema/body-composition.ts (ou adicionar em medical-profiles.ts)
export const bodyCompositionHistory = pgTable('body_composition_history', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id').notNull().references(() => users.id),
  documentId: uuid('document_id').references(() => documents.id),
  weight: numeric('weight', { precision: 5, scale: 2 }),
  bodyFat: numeric('body_fat', { precision: 5, scale: 2 }),
  muscleMass: numeric('muscle_mass', { precision: 5, scale: 2 }),
  visceralFat: numeric('visceral_fat', { precision: 5, scale: 2 }),
  boneMass: numeric('bone_mass', { precision: 5, scale: 2 }),
  bmr: integer('bmr'),
  bodyWater: numeric('body_water', { precision: 5, scale: 2 }),
  measuredAt: date('measured_at').notNull(),
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (table) => [
  index('body_composition_history_user_date_idx').on(table.userId, table.measuredAt),
])
```

## Scope

**IN:** Schema changes (campos + tabela), extração de métricas, atualização de perfil + histórico, feedback visual
**OUT:** UI de evolução temporal (Story 11.3), edição manual de dados de composição, import de CSV

## Complexity

**L** (Large) — Schema change + nova tabela + extração com parsing de valores + UI feedback. ~4h estimado.

## Risks

- **MEDIUM:** Parsing de valores de bioimpedância é frágil — formatos variam entre aparelhos/labs. Mitigação: regex permissiva + valores nulos para campos não encontrados.
- **LOW:** `measuredAt` pode não existir no structuredData — usar data de upload como fallback.
