## Story

**Como** paciente,
**Quero** saber quais campos foram preenchidos automaticamente pelo InBody e quando,
**Para** ter confiança na origem dos meus dados e saber quando atualizar.

---

## Contexto

Story 15.3 (Done) entregou a tab "Composição" com `BodyCompositionSection` renderizado como `{children}` em `ProfileForm`. Os dados exibidos vêm exclusivamente de `bodyCompositionHistory` — portanto são 100% provenientes de processamento InBody.

Esta story adiciona três camadas de UX sobre essa infraestrutura:
1. **Indicador de origem** — sinaliza visualmente que os dados vêm de bioimpedância
2. **Data da medição** — já existe no header da section mas precisa ser mais explícita e verificar validade (AC3)
3. **Estado vazio melhorado** — CTA acionável para upload quando não há dados

Não há mudanças de schema ou queries necessárias — toda a lógica é puramente de apresentação.

**Depends On:** Story 15.3 (Done — BLA-93)

---

## Acceptance Criteria

### AC1 — Badge de origem InBody nos dados de composição corporal

**Given** a tab "Composição" com dados InBody presentes
**When** o paciente visualiza o `BodyCompositionSection`
**Then** o header da seção exibe badge "InBody" ou ícone ⚡ indicando que os dados são provenientes de bioimpedância
**And** o badge deve ser visualmente distinto (ex: pill `text-xs bg-primary/10 text-primary rounded-full`)
**And** não deve ocupar espaço excessivo no mobile (390px)

### AC2 — Data da última medição InBody explícita no topo da tab

**Given** a tab "Composição" com ao menos uma medição InBody
**When** o paciente acessa a tab
**Then** a data do `measuredAt` mais recente de `bodyCompositionHistory` é exibida claramente no header do `BodyCompositionSection`
**And** o formato é por extenso: `"Última medição: 16 abr. 2026"` (pt-BR, usando `formatDate()` existente)
**And** a data aparece como texto de apoio — não substitui o header "Composição Corporal"

> **Nota de implementação:** `measuredAt` já é renderizado como `<span className="text-xs text-muted-foreground shrink-0">{formatDate(latest.measuredAt)}</span>`. AC2 requer tornar o label mais explícito: `"Última medição: {data}"` em vez de só a data.

### AC3 — Aviso visual quando dados InBody têm mais de 30 dias

**Given** a tab "Composição" com dados InBody presentes
**When** `measuredAt` do registro mais recente é anterior a 30 dias da data atual
**Then** exibe banner de aviso com token `--warning`: `"Dados desatualizados — faça nova bioimpedância"`
**And** o banner deve ter ícone AlertTriangle (Lucide) e link para `/app/upload`
**And** quando `measuredAt` tem ≤ 30 dias, o banner NÃO é exibido

> **Cálculo:** `daysSince = Math.floor((Date.now() - new Date(latest.measuredAt + 'T00:00:00').getTime()) / 86_400_000)`. Se `daysSince > 30` → exibir aviso. O cálculo ocorre no server component (não requer `Date.now()` client-side).

### AC4 — Estado vazio com CTA acionável

**Given** a tab "Composição" sem nenhum registro em `bodyCompositionHistory`
**When** o paciente acessa a tab
**Then** o estado vazio exibe botão/link "Enviar Bioimpedância" navegando para `/app/upload`
**And** o CTA usa componente `Link` do Next.js com estilo de botão primário
**And** a mensagem de contexto é mantida: "Envie um exame de bioimpedância para ver seus dados de composição corporal"

</span>

### Cálculo de staleness — AC3

```tsx
// No server component, após obter `latest`:
const daysSince = Math.floor(
  (Date.now() - new Date(latest.measuredAt + 'T00:00:00').getTime()) / 86_400_000
)
const isStale = daysSince > 30
```

Banner de aviso (condicional):
```tsx
{isStale && (
  <div className="flex items-center gap-2 rounded-md bg-warning/10 px-3 py-2 text-warning text-xs">
    <AlertTriangle className="size-3.5 shrink-0" aria-hidden="true" />
    <span>Dados desatualizados — </span>
    <Link href="/app/upload" className="underline font-medium">
      faça nova bioimpedância
    </Link>
  </div>
)}
```

Import: `import { AlertTriangle } from 'lucide-react'` e `import Link from 'next/link'` (já presente).
