## Story

**Como** sistema,
**Quero** índices nas foreign keys mais consultadas,
**Para** que queries de listagem e lookup tenham performance previsível com o crescimento de dados.

---

## Contexto

Atualmente só existem 3 índices custom: `snapshots(userId, createdAt)`, HNSW em `knowledge_embeddings.embedding` e GIN em `knowledge_embeddings.content_tsv`. Nenhum index existe em FKs de alta leitura como `documents.userId`, `analyses.userId`, `analyses.livingAnalysisVersionId`, `livingAnalysisVersions.livingAnalysisId` e `knowledgeBase.source`. Com dados reais, essas queries farão seq scan.

---

## Acceptance Criteria

### AC1 — Índice em `documents.userId`

**Given** a tabela `documents` sem index em `user_id`
**When** a migration é aplicada
**Then** existe index `documents_user_id_idx` em `documents(user_id)`
**And** `EXPLAIN` de `SELECT * FROM documents WHERE user_id = $1 ORDER BY created_at DESC` mostra Index Scan (não Seq Scan)

### AC2 — Índice em `analyses.userId`

**Given** a tabela `analyses` sem index em `user_id`
**When** a migration é aplicada
**Then** existe index `analyses_user_id_idx` em `analyses(user_id)`

### AC3 — Índice em `analyses.livingAnalysisVersionId`

**Given** a tabela `analyses` sem index em `living_analysis_version_id`
**When** a migration é aplicada
**Then** existe index `analyses_living_analysis_version_id_idx` em `analyses(living_analysis_version_id)`

### AC4 — Índice em `livingAnalysisVersions.livingAnalysisId`

**Given** a tabela `living_analysis_versions` sem index em `living_analysis_id`
**When** a migration é aplicada
**Then** existe index `living_analysis_versions_living_analysis_id_idx` em `living_analysis_versions(living_analysis_id)`

### AC5 — Índice em `knowledgeBase.source`

**Given** a tabela `knowledge_base` sem index em `source`
**When** a migration é aplicada
**Then** existe index `knowledge_base_source_idx` em `knowledge_base(source)`
**And** query de duplicata no auto-upload (`WHERE source = $1`) usa index

### AC6 — Migration sem downtime

**Given** banco com dados existentes (22 stories implementadas, docs/análises reais)
**When** `pnpm db:generate` e `pnpm db:migrate` são executados
**Then** migration gera apenas `CREATE INDEX` statements (não recria tabelas)
**And** dados existentes são preservados integralmente

### AC7 — Schema Drizzle atualizado

**Given** as tabelas `documents`, `analyses`, `livingAnalysisVersions`, `knowledgeBase`
**When** o schema é inspecionado
**Then** cada tabela tem callback `(table) => [index(...)]` com os novos índices
**And** `pnpm db:generate` não gera migration adicional (schema e DB em sync)

## Scope

**IN:** Criação de 5 índices em FKs de alta leitura, migration, verificação via EXPLAIN
**OUT:** Otimização de queries, CREATE INDEX CONCURRENTLY, índices compostos

## Complexity

**S** (Small) — Alteração puramente de schema, sem lógica de negócio. ~1h estimado.

---

## Risks

- **LOW:** Migration com `CREATE INDEX` em tabela com poucos registros é instantânea. Em produção com volume alto, considerar `CREATE INDEX CONCURRENTLY` (Drizzle não gera CONCURRENTLY por padrão — aceitável para o volume atual).
