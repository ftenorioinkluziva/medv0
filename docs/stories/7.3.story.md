## Story

**Como** sistema,
**Quero** um health endpoint para smoke test no deploy e política de senha forte para pacientes,
**Para** detectar falhas de conectividade no deploy e reduzir risco de contas comprometidas.

---

## Contexto

Não existe `/api/health`. O deploy atual (`sami.blackboxinovacao.com.br`) não tem smoke test automatizado. A política de senha é apenas `min(8)` sem exigência de complexidade — `registerSchema` e `resetPasswordSchema` em `src/lib/auth/validation.ts` usam `.min(8)` puro.

---

## Acceptance Criteria

### AC1 — Health endpoint

**Given** o endpoint `GET /api/health` não existe
**When** é criado e acessado
**Then** retorna JSON `{ status: 'ok', db: 'connected', timestamp: '<ISO>' }` com status 200

**Given** o banco está inacessível
**When** `GET /api/health` é chamado
**Then** retorna `{ status: 'degraded', db: 'disconnected', timestamp: '<ISO>' }` com status 503

**Given** o endpoint é público
**When** acessado sem autenticação
**Then** funciona normalmente (sem auth — é um smoke test)

### AC2 — DB ping real

**Given** o health endpoint precisa validar conectividade com Neon Postgres
**When** a verificação é executada
**Then** faz `SELECT 1` via Drizzle (`db.execute(sql\`SELECT 1\`)`)
**And** timeout de 5 segundos — se exceder, retorna `db: 'disconnected'`

### AC3 — Password policy

**Given** o `registerSchema` em `src/lib/auth/validation.ts` valida apenas `.min(8)`
**When** um paciente tenta registrar com senha `"abcdefgh"` (sem número/especial)
**Then** recebe erro: "Senha deve conter pelo menos 1 número ou caractere especial"

**Given** a nova validação
**When** paciente usa senha `"senha123"` (tem número)
**Then** a validação passa

**Given** a nova validação
**When** paciente usa senha `"senha@bc"` (tem especial)
**Then** a validação passa

### AC4 — Password policy no reset também

**Given** o `resetPasswordSchema` em `src/lib/auth/validation.ts`
**When** a policy de complexidade é atualizada
**Then** aplica a mesma regra: 8+ chars + pelo menos 1 número ou caractere especial
**And** `loginSchema` NÃO é alterado (login deve aceitar qualquer senha para não bloquear senhas legadas)

### AC5 — Mensagem de erro em português

**Given** as mensagens de erro de validação
**When** a senha não atende à policy
**Then** mensagens são em pt-BR consistentes com as existentes:
  - Min length: "Senha deve ter no mínimo 8 caracteres"
  - Complexity: "Senha deve conter pelo menos 1 número ou caractere especial"

### AC6 — Testes

**Given** o health endpoint e a nova password policy
**When** testes são executados
**Then:**
  - Health endpoint: 200 com DB ok, 503 com DB down (mock)
  - Password: aceita `"abc1defg"`, aceita `"abc@defg"`, rejeita `"abcdefgh"`, rejeita `"abc"` (curta + sem especial)

## Dev Notes

### Health endpoint pattern

```typescript
// src/app/api/health/route.ts
import { NextResponse } from 'next/server'
import { db } from '@/lib/db/client'
import { sql } from 'drizzle-orm'

export async function GET() {
  try {
    await db.execute(sql`SELECT 1`)
    return NextResponse.json({
      status: 'ok',
      db: 'connected',
      timestamp: new Date().toISOString(),
    })
  } catch {
    return NextResponse.json({
      status: 'degraded',
      db: 'disconnected',
      timestamp: new Date().toISOString(),
    }, { status: 503 })
  }
}
```

### Password regex

```typescript
// Zod custom refinement
z.string()
  .min(8, 'Senha deve ter no mínimo 8 caracteres')
  .refine(
    (val) => /[\d\W]/.test(val),
    'Senha deve conter pelo menos 1 número ou caractere especial',
  )
```

### Middleware exclusion

`proxy.ts` matcher já exclui `api/auth` — `/api/health` NÃO está nas rotas protegidas porque o matcher exclui `api/*` via `(?!api/auth|...)`. Verificar se o matcher atual permite `/api/health` sem auth.

Matcher atual: `['/((?!api/auth|_next/static|_next/image|favicon.ico|manifest.json|icons).*)']`

`/api/health` será capturado pelo matcher e passará pelo auth middleware — precisa ser adicionado à exclusão: `(?!api/auth|api/health|...)`.
