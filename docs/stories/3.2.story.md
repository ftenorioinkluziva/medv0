## Story

**Quero** enviar artigos médicos processados para o sistema via API autenticada,
**Para** que a base de conhecimento seja atualizada automaticamente sem intervenção manual.

## Contexto

Integração com o bot n8n externo (rodando na Hetzner). O bot scrapa transcrições de YouTube via Apify, processa com LLM em Knowledge Cards, e faz `POST` para este endpoint. O endpoint recebe o artigo, chunka, gera embeddings em batch e persiste. Invisível para o paciente.

O workflow n8n real está documentado em `doc/workflow_example_RAG_bot.json`.

Complemento direto da Story 3.1 — as tabelas `knowledge_base` e `knowledge_embeddings` (com cascade delete) já existem.

## Acceptance Criteria

### AC1 — Autenticação por API key

- Header `x-api-key` obrigatório em todas as requests
- Valor comparado com `process.env.KNOWLEDGE_API_KEY`
- Ausente ou inválido → `401` com `{ error: "Unauthorized" }`
- `KNOWLEDGE_API_KEY=` adicionado ao `.env.example` (seção medV0/SAMI)

### AC2 — `POST /api/admin/knowledge/auto-upload`

- Route Handler em `src/app/api/admin/knowledge/auto-upload/route.ts`
- `export const maxDuration = 60`
- Payload validado com Zod:

```typescript
const KnowledgeArticleSchema = z.object({
  title: z.string().min(1),
  content: z.string().min(1),
  summary: z.string().optional(),
  source: z.string().optional(),
  author: z.string().optional(),
  published_date: z.string().optional(), // ISO date string → convertida para Date
  category: z.string().optional(),
  subcategory: z.string().optional(),
  tags: z.array(z.string()).optional(),
  language: z.string().default('pt-BR'),
})
```

- Payload inválido → `400` com detalhes de validação Zod
- Artigo com `content` > 500.000 chars → `413 Content Too Large`

### AC3 — Algoritmo de chunking

Função `chunkText()` em `src/lib/ai/rag/chunker.ts`:

```typescript
export function chunkText(
  text: string,
  chunkSize: number = 2000,
  overlap: number = 200,
): string[]
```

- Target: ~500 tokens ≈ ~2000 chars por chunk (heurística: 1 token ≈ 4 chars em pt-BR)
- Overlap: ~50 tokens ≈ ~200 chars (sliding window)
- Respeitar quebras de parágrafo quando possível — não cortar no meio de frase
- Conteúdo < 2000 chars → retorna array com 1 único chunk

### AC4 — Geração de embeddings em batch

- `embedMany()` do pacote `ai` com `google.textEmbeddingModel('text-embedding-004')`
- Todos os chunks do artigo processados em uma única chamada batch
- `GOOGLE_GENERATIVE_AI_API_KEY` já configurado no projeto

### AC5 — Upsert com renovação de embeddings

Lógica por `(title, source)`:

1. Buscar `knowledge_base` por `title` + `source`
2. **Se existir:** atualizar metadados do artigo + deletar embeddings antigos (via `db.delete(knowledgeEmbeddings).where(...)`) + inserir embeddings novos
3. **Se não existir:** inserir artigo + inserir embeddings
4. Tudo em transação única (`db.transaction`)

Resposta de sucesso (`200`):

```json
{
  "success": true,
  "articleId": "uuid",
  "chunksCreated": 5,
  "action": "created" | "updated"
}
```

### Drizzle — Upsert Pattern

[Source: `src/lib/db/client.ts` + Drizzle ORM docs]

```typescript
import { db } from '@/lib/db/client'
import { and, eq } from 'drizzle-orm'

// 1. Buscar duplicado
const existing = await db
  .select({ id: knowledgeBase.id })
  .from(knowledgeBase)
  .where(and(
    eq(knowledgeBase.title, payload.title),
    eq(knowledgeBase.source, payload.source ?? ''),
  ))
  .limit(1)

// 2. Transação: update/insert artigo + delete embeddings antigos + insert novos
await db.transaction(async (tx) => {
  let articleId: string
  if (existing.length > 0) {
    articleId = existing[0].id
    await tx.update(knowledgeBase).set({ ...fields, updatedAt: new Date() }).where(eq(knowledgeBase.id, articleId))
    await tx.delete(knowledgeEmbeddings).where(eq(knowledgeEmbeddings.articleId, articleId))
  } else {
    const [inserted] = await tx.insert(knowledgeBase).values({ ...fields }).returning({ id: knowledgeBase.id })
    articleId = inserted.id
  }
  // insert embeddings em batch
  await tx.insert(knowledgeEmbeddings).values(
    chunks.map((content, i) => ({
      articleId,
      chunkIndex: i,
      content,
      embedding: embeddings[i],
    }))
  )
})
```

**Nota:** `on delete cascade` em `knowledge_embeddings.articleId` — delete do artigo remove embeddings automaticamente. Mas nesta story optamos pelo upsert explícito (não delete do artigo) para preservar `usageCount` e `isVerified`.

### Code Quality Assessment

Implementação sólida e bem estruturada. Três arquivos novos coesos, lógica pura testada diretamente (`chunkText`), padrões de DB seguidos corretamente. Um bug de lógica de upsert foi corrigido durante a revisão; error handling adicionado à route.

### Refactoring Performed

- **File:** `src/lib/ai/rag/uploader.ts`
  - **Change:** Import `isNull` de `drizzle-orm`; substituir `eq(knowledgeBase.source, input.source ?? '')` por condicional `input.source != null ? eq(...) : isNull(...)`
  - **Why:** `NULL != ''` em SQL — artigos sem `source` nunca eram encontrados pelo duplicate check, causando criação de duplicatas ao invés de update. Violava diretamente AC5.
  - **How:** Comparação correta por `isNull()` quando source é omitido; `eq()` quando fornecido.

- **File:** `src/app/api/admin/knowledge/auto-upload/route.ts`
  - **Change:** Adicionado try/catch em torno de `upsertKnowledgeArticle()`
  - **Why:** Sem error handling, falhas de DB ou da API embedMany causariam exceções não capturadas, podendo expor stack traces em ambientes não-produção.
  - **How:** Catch loga o erro com prefixo `[auto-upload]` e retorna `{ error: 'Internal server error' }` com status 500.

- **File:** `tests/unit/knowledge-auto-upload.test.ts`
  - **Change:** Adicionado teste "retorna 500 quando upsertKnowledgeArticle lança erro"
  - **Why:** Cobrir o novo path de error handling adicionado à route.
  - **How:** `mockRejectedValue(new Error(...))` valida status 500 e body correto.
