# Story 2.3 — Persistência do Snapshot

## Status: Ready for Review

## Story

**Como** paciente,
**Quero** que meus dados extraídos sejam salvos com data e hora,
**Para** que eu possa comparar a evolução dos meus exames ao longo do tempo.

---

## Contexto

Persiste o resultado sanitizado da extração Gemini. Dois registros em transação única: `documents` (metadados) + `snapshots` (JSONB completo). O arquivo original já foi descartado antes desta etapa. O snapshot é a fonte de verdade para análises futuras.

**Ponto de integração:** o route handler `src/app/api/documents/upload/route.ts` já tem o `TODO: Story 2.3` aguardando implementação. BLA-42 já entrega `SanitizedMedicalDocument` via `extractMedicalDocument()`.

---

## Acceptance Criteria

### AC1 — Schema `documents`
- Tabela `documents` com colunas:
  - `id` (UUID, PK, defaultRandom)
  - `user_id` (FK → users, not null)
  - `document_type` (text — ex: "Exame de Sangue", "Hemograma")
  - `original_file_name` (text — apenas para display, sem conteúdo)
  - `exam_date` (date, nullable — extraída do Gemini quando disponível)
  - `extracted_at` (timestamp, not null — momento da extração)
  - `overall_summary` (text, nullable)
  - `processing_status` (text enum: `processing` | `completed` | `failed`, default `completed`)
  - `processing_error` (text, nullable)
  - `created_at` (timestamp, default now)
  - `updated_at` (timestamp)

### AC2 — Schema `snapshots`
- Tabela `snapshots` com colunas:
  - `id` (UUID, PK, defaultRandom)
  - `document_id` (FK → documents, not null, unique — relação 1:1)
  - `user_id` (FK → users, not null — desnormalizado para queries diretas)
  - `structured_data` (jsonb, not null — `SanitizedMedicalDocument`)
  - `created_at` (timestamp, default now)
- Tipagem Drizzle: `structuredData: jsonb('structured_data').$type<SanitizedMedicalDocument>()`
- Índice: `snapshots(user_id, created_at DESC)` para listagem de histórico

### AC3 — Transação de persistência
- Insert `documents` + insert `snapshots` em transação única (`db.transaction`)
- Rollback automático se qualquer insert falhar
- `exam_date` extraída de `structuredData.examDate` (quando disponível, senão `null`)
- Resposta ao usuário inclui `documentId` para navegação pós-upload

### AC4 — Isolamento por usuário
- `user_id` propagado em ambas as tabelas a partir de `session.user.id`
- Query de listagem sempre filtra por `userId = session.user.id`
- Snapshot de outro usuário retorna 404 (não 403 — não revelar existência)

### AC5 — Arquivo original descartado
- Nenhuma referência ao buffer original após persistência
- Zero storage de arquivo: sem Blob, S3 ou disco local

### AC6 — Integração com route handler
- `src/app/api/documents/upload/route.ts` chamar `extractMedicalDocument()` + `persistSnapshot()` em sequência
- Substituir os dois TODOs existentes (Story 2.2 e Story 2.3)
- Resposta final: `{ success: true, documentId: string, fileName: string }`

## Notas Técnicas

- **Drizzle transação:** `await db.transaction(async (tx) => { await tx.insert(documents)...; await tx.insert(snapshots)... })`
- **JSONB type-safe:** `jsonb('structured_data').$type<SanitizedMedicalDocument>()` — importar type de `@/lib/documents/extractor`
- **exam_date:** tipo `date` (não `timestamp`) — apenas data do exame sem hora
- **Schema import pattern:** sempre via `import { table } from '@/lib/db/schema'` (barrel export)
- **processing_status enum:** definir como `pgEnum` no Drizzle para validação no DB
