## Story

**Como** sistema,
**Quero** associação many-to-many entre agentes e artigos da knowledge base,
**Para** que cada agente busque RAG filtrada pelo seu escopo de conhecimento.

---

## Contexto

Atualmente `searchKnowledge(query, topK)` busca em TODA a base sem filtro de agente. Um agente de exercícios recebe chunks sobre endocrinologia. A segmentação requer:
1. Nova tabela `agent_knowledge` (many-to-many)
2. Campo `isGlobal` em `knowledge_base` para artigos acessíveis por todos
3. `searchKnowledge` aceitar `agentId` opcional para filtrar

**Schema atual relevante:**
- `knowledge_base` — sem `isGlobal`
- `knowledge_embeddings` — FK `articleId` → `knowledge_base.id`
- `health_agents` — sem relação com knowledge
- `searchKnowledge()` em `src/lib/ai/rag/vector-search.ts` — assinatura `(query, topK=5)`

---

## Acceptance Criteria

### AC1 — Tabela `agent_knowledge`

**Given** não existe relação entre agentes e artigos
**When** a migration é aplicada
**Then** existe tabela `agent_knowledge` com:
  - `id` UUID PK
  - `agentId` UUID FK → `health_agents.id` ON DELETE CASCADE
  - `articleId` UUID FK → `knowledge_base.id` ON DELETE CASCADE
  - `createdAt` TIMESTAMP DEFAULT NOW
  - UNIQUE constraint em `(agentId, articleId)`

### AC2 — Campo `isGlobal` em `knowledge_base`

**Given** tabela `knowledge_base` sem campo `isGlobal`
**When** a migration é aplicada
**Then** campo `isGlobal` BOOLEAN DEFAULT TRUE existe
**And** todos os artigos existentes têm `isGlobal = true` (backward compatibility)

### AC3 — `searchKnowledge` com filtro por agente

**Given** `searchKnowledge(query, topK)` busca global
**When** assinatura é atualizada para `searchKnowledge(query, topK, agentId?)`
**Then:**

**Cenário A — sem agentId (backward compat):**
**Given** `searchKnowledge('diabetes', 5)` sem agentId
**When** executado
**Then** busca em TODOS os embeddings (comportamento atual preservado)

**Cenário B — com agentId, artigo associado:**
**Given** agente "Nutrição" associado a artigo "Dieta Mediterrânea" via `agent_knowledge`
**When** `searchKnowledge('dieta', 5, nutriçãoAgentId)`
**Then** retorna chunks do artigo "Dieta Mediterrânea"

**Cenário C — com agentId, artigo global:**
**Given** artigo "Bases de Medicina Integrativa" com `isGlobal = true`, não associado a nenhum agente
**When** `searchKnowledge('integrativa', 5, nutriçãoAgentId)`
**Then** retorna chunks do artigo global (acessível por todos)

**Cenário D — com agentId, artigo de outro agente:**
**Given** artigo "Protocolos de Exercício" associado apenas ao agente "Exercício" e `isGlobal = false`
**When** `searchKnowledge('exercício', 5, nutriçãoAgentId)`
**Then** NÃO retorna chunks desse artigo

### AC4 — Queries helper

**Given** a nova relação agent-knowledge
**When** queries são necessárias
**Then** existem em `src/lib/db/queries/knowledge.ts`:
  - `getArticlesByAgent(agentId): KnowledgeBase[]`
  - `associateArticleToAgent(agentId, articleId): void`
  - `disassociateArticleFromAgent(agentId, articleId): void`
  - `getAgentsByArticle(articleId): HealthAgent[]`

### AC5 — `analyzeWithAgent` usa filtro

**Given** `analyzeWithAgent()` em `src/lib/ai/agents/analyze.ts` chama `searchKnowledge` sem agentId
**When** atualizado para passar `agent.id` como `agentId`
**Then** RAG search é filtrada pelo escopo do agente
**And** agentes sem artigos associados ainda recebem artigos globais

### AC6 — Migration preserva dados

**Given** banco com artigos e embeddings existentes
**When** migration é aplicada
**Then** todos os artigos existentes ficam com `isGlobal = true`
**And** tabela `agent_knowledge` começa vazia (associações feitas via admin)
**And** nenhum dado é perdido

### AC7 — Testes

**Given** as mudanças
**When** testes são executados
**Then:**
  - Schema: `agent_knowledge` com unique constraint funciona (insert + duplicate rejection)
  - `searchKnowledge` sem agentId: comportamento idêntico ao anterior
  - `searchKnowledge` com agentId: filtra por associação + global
  - `analyzeWithAgent`: passa `agent.id` ao RAG search
  - Queries helper: CRUD de associações

## Dev Notes

### Schema `agent_knowledge`

```typescript
// src/lib/db/schema/knowledge.ts
import { healthAgents } from './health-agents'

export const agentKnowledge = pgTable('agent_knowledge', {
  id: uuid('id').primaryKey().defaultRandom(),
  agentId: uuid('agent_id')
    .notNull()
    .references(() => healthAgents.id, { onDelete: 'cascade' }),
  articleId: uuid('article_id')
    .notNull()
    .references(() => knowledgeBase.id, { onDelete: 'cascade' }),
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (table) => [
  unique('agent_knowledge_agent_article_unique').on(table.agentId, table.articleId),
  index('agent_knowledge_agent_id_idx').on(table.agentId),
  index('agent_knowledge_article_id_idx').on(table.articleId),
])
```

### `isGlobal` em `knowledge_base`

```typescript
// Adicionar campo
isGlobal: boolean('is_global').notNull().default(true),
```

### Filtro em `searchKnowledge` — approach SQL

Quando `agentId` é fornecido, ambas as queries (vector e BM25) precisam de um WHERE adicional:

```sql
WHERE ke.article_id IN (
  SELECT ak.article_id FROM agent_knowledge ak WHERE ak.agent_id = $agentId
  UNION
  SELECT kb.id FROM knowledge_base kb WHERE kb.is_global = true
)
```

Em Drizzle, usar `inArray` com subquery ou raw SQL.

### Backward compat

Sem `agentId` → query sem filtro → comportamento idêntico ao v1.0.

## Scope

**IN:** Tabela `agent_knowledge`, campo `isGlobal`, filtro em `searchKnowledge`, queries helper, update em `analyzeWithAgent`
**OUT:** Admin UI de associação (Story 8.2), migration de artigos existentes para agentes (feita via admin)

## Complexity

**L** (Large) — Schema + migration + query refactor (vector + BM25 com subquery) + update no pipeline. ~4h estimado.

## Risks

- **MEDIUM:** Filtro por agente nas queries de busca pode impactar performance se subquery for lenta. Mitigação: índices em `agent_knowledge(agentId)` e `agent_knowledge(articleId)`.
- **LOW:** Backward compat — sem `agentId`, busca global. Testes cobrem cenário.
